# Bad Cases

## Bad Case 1：字幕提取失败

### 现象

视频没有字幕，或者 yt-dlp 无法解析视频平台返回的数据。

### 可能原因

视频本身没有字幕；平台限制访问；网络或代理异常；yt-dlp 版本不兼容。

### 当前解决方案

先走 subtitle-first，如果字幕提取失败，再进入 Whisper fallback。

### 后续优化方向

记录字幕语言、字幕格式和平台错误码；增加 yt-dlp 版本提示；对可恢复错误增加重试。

### 面试说法

我通过 subtitle-first + Whisper fallback 处理字幕缺失问题，保证有字幕时低成本处理，无字幕时仍有兜底方案。

## Bad Case 2：Whisper 转写失败

### 现象

音频下载失败、模型加载失败、本地算力不足，或者转写结果为空。

### 可能原因

视频链接不可下载；磁盘空间不足；faster-whisper 模型未下载；CPU 转写耗时过长；音频质量差。

### 当前解决方案

返回明确 error，记录 trace 中的 `fallback_used`、`audio_downloaded`、`whisper_model_size` 和 `latency_ms`。

### 后续优化方向

接入异步任务队列，增加任务状态查询；支持更小模型或 GPU 配置；增加音频清理策略。

### 面试说法

Whisper 是兜底能力，不应该让接口静默失败。我会把下载、模型和转写错误都写入 trace，方便定位瓶颈。

## Bad Case 3：LLM JSON 格式错误

### 现象

模型返回 Markdown、解释文字或不完整 JSON，导致解析失败。

### 可能原因

Prompt 约束不够强；模型输出不稳定；长上下文导致指令被稀释。

### 当前解决方案

加强 Prompt 约束，要求只返回 JSON；在 `core/llm.py` 中增加 JSON 代码块清理和 JSON 片段提取；失败时记录错误。

### 后续优化方向

增加 schema 校验和自动修复重试；对关键链路使用更稳定的结构化输出能力。

### 面试说法

Agent 项目不能假设 LLM 永远稳定输出，我在 LLM 封装层做了 JSON 解析兜底，并把失败写入 trace 方便复盘。

## Bad Case 4：思维导图格式错误

### 现象

Mermaid mindmap 缩进错误、包含多余解释文字，或者没有以 `mindmap` 开头。

### 可能原因

模型没有严格遵守输出格式；输入文本结构混乱；prompt 示例不够明确。

### 当前解决方案

在 mindmap skill 中要求只返回 JSON，并要求 `mindmap` 字段中的内容以 `mindmap` 开头。

### 后续优化方向

增加 `mindmap_validator`，对 Mermaid 文本做格式检查；失败时让模型重试修复。

### 面试说法

思维导图是结构化展示结果，后续我会增加 validator，把生成和校验拆开，避免前端渲染时才发现格式问题。

## Bad Case 5：长文本总结丢信息

### 现象

长视频或长 transcript 经过压缩后，部分细节、例子或上下文被丢失。

### 可能原因

chunk 太大或太小；overlap 不合适；map-reduce 合并时过度压缩；模型忽略局部细节。

### 当前解决方案

使用 chunk + map-reduce summary，并在 trace 中记录 `chunk_count`、`compression_used`、`compressed_length`。

### 后续优化方向

保留 chunk 级摘要；为重要术语建立索引；对用户问题相关片段做二次召回。

### 面试说法

长文本压缩不是简单截断，我用分段总结再合并的方式降低上下文压力，同时把压缩元信息写入 trace 便于分析信息损失。

## Bad Case 6：后端接口超时

### 现象

视频处理、音频下载、Whisper 转写或 LLM 调用耗时长，前端等待时间久。

### 可能原因

同步接口处理重任务；网络慢；模型推理慢；本地 CPU 转写耗时过长。

### 当前解决方案

第一版使用同步接口，前端设置 timeout 并展示友好错误；后端 trace 记录 `latency_ms`。

### 后续优化方向

引入任务队列和任务状态查询；将视频处理改成异步任务；前端轮询任务状态。

### 面试说法

MVP 先用同步接口跑通闭环，但我已经通过 trace 记录耗时。后续如果进入生产形态，会把视频处理拆成异步任务。

## Bad Case 7：GitHub / 外部网络访问失败

### 现象

访问 GitHub、YouTube、字幕 URL 或模型 API 超时。

### 可能原因

本地网络、代理、平台访问限制或模型服务不可用。

### 当前解决方案

记录为环境问题，保留错误信息，不把外部网络失败归因于业务代码。

### 后续优化方向

增加网络诊断说明；对模型 API 和下载接口增加重试、超时和降级策略。

### 面试说法

外部依赖失败是 Agent 应用的常见风险，我会把这类错误显式记录在 trace 中，并区分环境问题和业务逻辑问题。
